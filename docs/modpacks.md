# Modpacks And Mods

This guide explains how to use modpacks and standalone mods with Audentity Runner.

Audentity Runner supports multiple workflows because not every pack is distributed in the same format. Some packs are published on CurseForge or Modrinth and can be imported automatically. Others are exported archives or prebuilt server packs that must be placed manually in the data directory.

## Quick Decision Guide

Use this rule of thumb before touching `.env`:

1. If the modpack is publicly published on CurseForge, use the CurseForge workflow.
2. If the modpack is publicly published on Modrinth, use the Modrinth workflow.
3. If you have an exported `.mrpack`, use the local Modrinth file workflow.
4. If you have an exported CurseForge `.zip` with a proper CurseForge manifest, use the local CurseForge archive workflow.
5. If you already have a ready-to-run server pack that contains folders like `mods/`, `config/`, and `datapacks/`, extract it directly into `server/data` and use the manual server pack workflow.
6. If you do not have a full modpack and only want to add a few mods, use the loose mods workflow.

## Core Concepts

### `TYPE`

`TYPE` selects the server loader used by `itzg/minecraft-server`.

Common values:

- `VANILLA`
- `FABRIC`
- `FORGE`
- `NEOFORGE`

Your chosen pack must match the loader.

Examples:

- Fabric pack -> `TYPE=FABRIC`
- Forge pack -> `TYPE=FORGE`
- NeoForge pack -> `TYPE=NEOFORGE`

### `VERSION`

`VERSION` is the Minecraft version used by the server container.

It must match the pack.

Examples:

- Fabric 1.20.1 pack -> `VERSION=1.20.1`
- NeoForge 1.21.1 pack -> `VERSION=1.21.1`

### `LEVEL`

`LEVEL` controls the world folder name inside `server/data`.

Use a different `LEVEL` when testing a new pack so you do not risk your main save.

Example:

```env
LEVEL=modpack-test-1
```

### `server/modpacks`

Use `server/modpacks` for local archive files that should be mounted into the container as `/modpacks/...`.

Typical examples:

- local CurseForge `.zip`
- local Modrinth `.mrpack`

### `server/data`

`server/data` is the persistent Minecraft server directory.

This is where the running server keeps:

- `mods/`
- `config/`
- `world/`
- `logs/`
- generated server files

Some workflows import content into `server/data` automatically. Others require you to extract files there yourself.

## Workflow A: Public CurseForge Modpack

Use this when the modpack is publicly available on CurseForge and you want the Docker image to install it automatically.

### Required `.env` values

```env
TYPE=FORGE
VERSION=1.20.1
MODPACK_PLATFORM=AUTO_CURSEFORGE
CF_API_KEY=your_api_key_here
CF_SLUG=pack-slug
```

Notes:

- `CF_SLUG` is the modpack slug from the CurseForge URL.
- `CF_API_KEY` is required by the CurseForge API because the automatic CurseForge workflow does not use a pre-extracted server pack. Instead, the container asks CurseForge for modpack metadata and download information so it can resolve and install the pack during startup.
- Create the key in the CurseForge Console: [https://console.curseforge.com/](https://console.curseforge.com/)
- In practice, Audentity Runner only needs `CF_API_KEY` for CurseForge-based automatic import flows. It is not needed for public Modrinth packs, local `.mrpack` files, manual server packs extracted into `server/data`, or loose standalone mods.
- If the API key contains `$`, write each dollar sign as `$$` in `.env`.

Example:

```env
CF_API_KEY=$$2a$$10$$example
```

### When this workflow is correct

Use it when:

- the pack is published on CurseForge
- the pack is meant to be fetched from CurseForge
- you want Docker to install and resolve the pack automatically

### When this workflow is wrong

Do not use it when:

- you already have a ready-made server pack zip
- the archive already contains the final `mods/`, `config/`, and `datapacks/` directories
- the zip was generated by another tool and is not a native CurseForge import archive

## Workflow B: Public Modrinth Modpack

Use this when the modpack is publicly available on Modrinth.

### Required `.env` values

```env
TYPE=FABRIC
VERSION=1.20.1
MODPACK_PLATFORM=MODRINTH
MODRINTH_MODPACK=pack-slug-or-url
```

Notes:

- `MODRINTH_MODPACK` can be a slug, project ID, or full URL.
- No API key is required.

### When this workflow is correct

Use it when:

- the pack is publicly published on Modrinth
- you want Docker to fetch it automatically

## Workflow C: Local Modrinth `.mrpack`

Use this when you already downloaded a Modrinth pack file.

### Steps

1. Put the `.mrpack` file inside `server/modpacks`.
2. Point `.env` to the mounted file path.

### Required `.env` values

```env
TYPE=FABRIC
VERSION=1.20.1
MODPACK_PLATFORM=MODRINTH
MODRINTH_MODPACK=/modpacks/name-of-your-pack.mrpack
```

### When this workflow is correct

Use it when:

- you exported or downloaded a `.mrpack`
- you want the Docker image to import it automatically

## Workflow D: Local CurseForge Archive

Use this when you have a local CurseForge archive that is still meant to be processed by the CurseForge import flow.

### Steps

1. Put the `.zip` file inside `server/modpacks`.
2. Point `.env` to the mounted file path.

### Required `.env` values

```env
TYPE=FORGE
VERSION=1.20.1
MODPACK_PLATFORM=AUTO_CURSEFORGE
CF_API_KEY=your_api_key_here
CF_SLUG=custom
CF_MODPACK_ZIP=/modpacks/name-of-your-pack.zip
```

### Important requirements

- `CF_API_KEY` is still required because the container still uses the CurseForge API even though the archive file is local.
- The archive must be a CurseForge-style import archive.
- The archive must contain a valid CurseForge manifest that the image can parse.

### Common failure cases

- The key in `.env` uses `$` instead of `$$`.
- The zip is not actually a valid zip.
- The zip has the wrong internal structure.
- The zip is a ready-made server pack rather than a CurseForge import archive.

## Workflow E: Manual Server Pack Extraction

Use this when the pack is already a server-ready bundle.

This is the correct workflow for packs that already contain things like:

- `mods/`
- `config/`
- `datapacks/`
- helper scripts such as `start.bat` or `start.sh`

These packs should not go through `AUTO_CURSEFORGE`.

### Steps

1. Set the correct loader and version in `.env`.
2. Remove or disable modpack import variables.
3. Extract the server pack contents directly into `server/data`.
4. Start the stack.

### Required `.env` values

Example for a Fabric 1.20.1 server pack:

```env
TYPE=FABRIC
VERSION=1.20.1
```

### Variables that should not stay active

Disable or remove these if you are using a manual server pack:

```env
MODPACK_PLATFORM=
CF_API_KEY=
CF_SLUG=
CF_MODPACK_ZIP=
MODRINTH_MODPACK=
```

### Correct extraction result

After extraction, `server/data` should contain the pack content directly.

Correct:

```text
server/data/
  mods/
  config/
  datapacks/
  manifest.json
```

Wrong:

```text
server/data/
  MyPack/
    mods/
    config/
```

### Why this matters

If you extract into an extra nested folder, the server will not see the pack files where it expects them.

### About the issue we hit during setup

One class of server packs looks like a zip with `mods/`, `config/`, and a `manifest.json`, but that manifest is not a CurseForge import manifest. In that case, the correct fix is not to keep fighting the CurseForge installer. The correct fix is to extract the pack into `server/data` and run it as a manual server pack with the right `TYPE` and `VERSION`.

## Workflow F: Loose Standalone Mods

Use this when you do not have a full modpack and only want to add individual mods.

### Steps

1. Choose the correct loader in `.env`.
2. Start the server once so `server/data` exists.
3. Put `.jar` files into `server/data/mods`.
4. Restart the server.

### Required `.env` values

Example:

```env
TYPE=FABRIC
VERSION=1.20.1
```

### Rules

- Only use mods compatible with your loader and Minecraft version.
- Client-only mods usually do not belong on the server.
- Some mods require dependency mods such as libraries or APIs.

## How To Choose The Correct Loader

Use the modpack's own metadata, page, or included documentation.

Typical signals:

- `fabric` in jar names -> use `TYPE=FABRIC`
- `forge` in jar names -> use `TYPE=FORGE`
- `neoforge` in docs or release notes -> use `TYPE=NEOFORGE`

If the pack includes many files like:

- `fabric-api`
- `voicechat-fabric`
- `*-fabric-*.jar`

then it is almost certainly a Fabric pack.

## Recommended Safe Testing Flow

When changing packs:

1. Pick the correct workflow from this guide.
2. Set a temporary `LEVEL` for a test world.
3. Start the server and watch logs.
4. Join the server and confirm mods loaded correctly.
5. Only then move the setup to your main world if needed.

Example:

```env
LEVEL=pack-validation-1
```

## Troubleshooting

### The container restarts immediately

Check:

```bash
docker compose --env-file .env -f server/docker-compose.yml logs --tail=200 minecraft
```

### CurseForge says the API key is invalid

If your key contains dollar signs, escape them in `.env` with `$$`.

Example:

```env
CF_API_KEY=$$2a$$10$$example
```

### CurseForge import fails on a local zip

Possible reasons:

- the archive is not a CurseForge import archive
- the archive is a manual server pack instead
- the file is corrupted
- the file path in `CF_MODPACK_ZIP` is wrong

If the archive already contains the final server files, switch to the manual server pack workflow.

### The server starts but mods are missing

Check:

- the selected `TYPE`
- the Minecraft `VERSION`
- whether the mods were extracted into `server/data/mods`
- whether dependencies are present

### I want to keep multiple worlds for different packs

Use different `LEVEL` values so each pack can have its own world folder.
